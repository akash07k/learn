# learn

An accessible learning corpus. `learn` collects self-contained subjects, each teaching one area
entirely through text, commands, and the keyboard, written for a reader who is blind and uses a
screen reader. Accessibility is the reason the project exists: every guide avoids the graphical UI,
names exact commands and file paths, and is published as accessible HTML whose heading tree is a
navigable table of contents.

The corpus is modular and extensible. Each subject lives in its own folder under `subjects/` with
its own guides, glossary, and configuration, and the shared tooling discovers subjects
automatically, so adding one never touches the tooling.

## Subjects

- [Proxmox VE Zero-to-Hero](subjects/proxmox/guides/README.md) - running Proxmox VE entirely from
  the shell on a single home node, for a blind operator who SSHes in from a separate control
  station. The flagship subject.
- [Subject Template](subjects/_template/guides/README.md) - the minimal skeleton to copy when
  starting a new subject.

## Repository layout

- `subjects/<name>/` - one self-contained subject. Holds a `subject.toml` manifest, a `CONTEXT.md`
  glossary of roles, `guides/` (the numbered guides, plus any cheatsheets and recipes), and
  optionally `docs/adr/` and `research/`. A subject may add an `AUTHORING-NOTES.md` for its
  domain-specific conventions.
- `AUTHORING-CONVENTIONS.md` - the subject-agnostic conventions every guide follows.
- `ADDING-A-SUBJECT.md` - how to add a new subject.
- `MAINTAINING.md` - maintenance rules for gates, local scratch files, ADR discipline, upstream
  drift, naming, and releases.
- `assets/` - the shared pandoc template, accessibility CSS, and link filter used to build every
  subject.
- `tools/` - the build and check tooling: `subjects.ts` (discovery), `build-html.ts` (pandoc build),
  `checks.ts` (the glyph, link, and convention checks), and `check.ts` (the local quality gate that
  ties it all together; run it with `bun run check`).
- `docs/agents/` - local agent workflow conventions.
- `.scratch/` - gitignored local markdown issue tracker used by agent workflows. See
  [AGENTS.md](AGENTS.md) and [docs/agents/issue-tracker.md](docs/agents/issue-tracker.md).

## Requirements

- Bun 1.3.14 or newer in the 1.x line. CI currently uses Bun 1.3.14.
- Pandoc 3.1.11.1. CI pins this version because pandoc output can change between releases.

Install Bun from [bun.sh](https://bun.sh/) and pandoc from the
[pandoc installing guide](https://pandoc.org/installing.html). Then verify both are on your PATH:

```bash
bun --version
pandoc --version
```

Install the JavaScript and TypeScript toolchain from the lockfile:

```bash
bun install --frozen-lockfile
```

That also runs the repository's `prepare` script, which installs the Lefthook git hooks.

## Building and checking

The guides are published as accessible standalone HTML via pandoc. From the repository root, build
every subject with:

```bash
bun run build
```

The output is written to `html/` (gitignored), one tree per subject under `html/<subject>/`, plus an
`html/index.html` landing page. To run the quality gate - the build, plus a banned-glyph scan, an
internal-link check, and a per-subject convention lint - use:

```bash
bun run check
```

That command does not run every CI check. To reproduce GitHub Actions locally, use the CI-parity
gate:

```bash
bun run ci:local
```

`ci:local` runs JavaScript linting, formatter checks, Markdown linting, TypeScript typechecking,
tests, and then `bun run check`.

The build and checks are TypeScript run on Bun (in `tools/`); pandoc is the only external tool. The
gate fails if any guide contains a glyph a screen reader cannot read cleanly, if any internal link
or `#fragment` does not resolve in the built HTML, or if a convention check trips (a broken
Previous/Next footer chain, an ascii `->` arrow, or a subject-defined lint rule). The glyph scan
covers subject `content_dirs` plus a small fixed set of repository-root docs. The convention check
covers each subject's `lint_dirs`. Markdownlint and the Markdown formatter check tracked Markdown
files across the repository, excluding ignored output such as `html/` and `tmp/`.

On Windows, `run.bat` offers the same actions from a plain numbered menu.

## Generated outputs

- `html/` is generated by `bun run build` and ignored by git. It is safe to delete and rebuild.
- `tmp/` is ignored by git and used for scratch files, local handoff notes, and temporary tool
  output. Do not rely on it for durable project documentation.
- `.scratch/` is ignored by git and used for the local markdown issue tracker. It is safe to create
  locally when agent workflows need issue or PRD files.

## Commits and hooks

The repository uses Lefthook for local checks and commitlint for Conventional Commit messages. A
normal `bun install --frozen-lockfile` installs the hooks through the `prepare` script; if you need
to reinstall them manually, run:

```bash
bun run prepare
```

Commit signing is not required by the repository tooling. Use the signing policy for your own
environment; commitlint enforces message shape, not signatures.

Commit types follow the conventional set (`docs`, `fix`, `test`, `build`, `ci`, and so on). Three
repo-specific types are also allowed: `scaffold` for subject/template scaffolding, `subjects` for
multi-subject corpus changes, and `tooling` for build/check tooling changes.

## Adding a subject

Copy `subjects/_template/` to `subjects/<your-subject>/`, edit its `subject.toml`, and write your
guides. The tooling picks it up automatically. See `ADDING-A-SUBJECT.md` for the full walkthrough.
